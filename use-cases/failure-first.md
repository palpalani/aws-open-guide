# Failure-first patterns

> The patterns every playbook's "Failure modes" section refers to. AWS architectures are designed for failure tolerance; this is the shared vocabulary.

When a playbook says "exponential backoff with jitter" or "use a DLQ," it's pointing here. Each pattern below is service-agnostic enough to apply across most workloads, with the AWS-specific knobs called out.

---

## 1. Retries with exponential backoff and jitter

**The problem:** something downstream is throttled or briefly unavailable. Naive retry storms make it worse — every client retries at the same time and DDoS the recovering service.

**The pattern:**
- Wait an exponentially increasing duration between retries: 100ms, 200ms, 400ms, 800ms, …
- Add **jitter** — a randomised component to spread retry attempts across time
- Cap total retry count and total elapsed time
- Distinguish retryable (5xx, throttle, network) from non-retryable (4xx auth, 4xx validation) errors

**AWS specifics:**
- The AWS SDKs implement this by default; the defaults are reasonable for most workloads
- Configure via `retryMode: 'adaptive'` in v3 SDKs for client-side rate limiting
- Lambda's built-in retry on async invocation: 2 retries, then DLQ — this is retry, not retry-with-backoff in your code
- Step Functions task retries: configure `Retry` block per state with `IntervalSeconds`, `BackoffRate`, `MaxAttempts`
- API Gateway → Lambda integration: 5xx returned to caller; client controls retry

**Anti-pattern:** retry-without-jitter retry storm; retrying a 4xx because "it might work next time"; no cap on total wait so a request hangs for minutes.

**Reference:** [Timeouts, retries, and backoff with jitter (Builders Library)](https://aws.amazon.com/builders-library/timeouts-retries-and-backoff-with-jitter/) is the canonical AWS write-up.

## 2. Idempotency

**The problem:** retries (yours or AWS's) cause the same operation to execute twice. Without idempotency you create two orders, charge a card twice, send two emails.

**The pattern:**
- Every state-changing request carries an **idempotency key** — a UUID generated by the client
- The server records `(idempotency_key, result)`; on a duplicate key, return the recorded result instead of re-executing
- The key has a TTL appropriate to your retry window (24 hours is typical)

**AWS specifics:**
- **DynamoDB conditional writes** — `ConditionExpression: 'attribute_not_exists(pk)'`; first writer wins
- **SQS message deduplication** — FIFO queues with `MessageDeduplicationId` (5-minute window)
- **EventBridge** — does not deduplicate; you must do it downstream
- **Stripe-style idempotency** is a request header for HTTP APIs; implement at API Gateway or in app
- **AWS Powertools for Lambda** has an idempotency utility that backs onto DynamoDB

**Where it's not optional:**
- Anything that costs money (charges, sends, refunds)
- Anything that's user-visible and unique (orders, registrations)
- SQS standard queues (at-least-once delivery means duplicates happen)
- Lambda async invocation (retried automatically)

**Anti-pattern:** "idempotent because it does the same thing each time" — that's not idempotent if it has side effects. True idempotency means the *side effect* happens at most once.

## 3. Dead-letter queues (DLQs)

**The problem:** a message can't be processed. You don't want to retry forever (poison message blocks the queue) or drop it (silent data loss).

**The pattern:**
- After N failed attempts (`maxReceiveCount` for SQS), move the message to a DLQ
- DLQ is a separate queue; nothing reads it on the happy path
- An alarm fires when the DLQ has any messages
- A redrive process (manual or automated) reprocesses DLQ messages after the upstream issue is fixed

**AWS specifics:**
- **SQS DLQ** — set `RedrivePolicy` on the source queue with `deadLetterTargetArn` and `maxReceiveCount` (typically 3–10)
- **SQS redrive** — built-in `StartMessageMoveTask` API or via the console
- **Lambda async DLQ** — `DeadLetterConfig` on the function (older) or **destinations** (newer, preferred — sends to SQS/SNS/EventBridge/Lambda on success or failure)
- **EventBridge** — `DeadLetterConfig` on the rule target; failed targets go to SQS
- **Step Functions** — no DLQ; failures go to a `Catch` block that can route to a recovery workflow

**Alarms:** `ApproximateNumberOfMessagesVisible > 0` on the DLQ. Page on it.

**Anti-pattern:** DLQ exists, no alarm. Or DLQ alarm exists, nobody on call knows what to do. Write the runbook before launching: "a DLQ message means X, here's how to inspect, here's how to redrive after fix."

## 4. Timeouts

**The problem:** without an explicit timeout, a slow downstream pins resources and cascades latency.

**The pattern:**
- Every network call has a timeout, set shorter than the caller's timeout
- Connection timeout vs read timeout — set both
- Timeouts compose: API Gateway 29s → Lambda 28s → DynamoDB call 5s → external API call 3s

**AWS specifics:**
- **API Gateway** — 29s hard maximum integration timeout
- **Lambda** — 15min hard limit; for Step Functions tasks, also has its own timeout
- **ALB** — 60s default idle timeout, configurable
- **AWS SDKs** — set `requestHandler` timeouts in v3; defaults are too long for production (often 100s+)
- **HTTP clients** — never use library defaults; explicit timeouts every time

**Cascading rule:** the caller's timeout must be larger than the sum of the callee's timeouts including retries. Otherwise the caller times out mid-retry and you've done extra work for nothing.

**Anti-pattern:** no timeout on database queries; relying on TCP keepalive to detect dead peers.

## 5. Circuit breakers

**The problem:** a downstream is sustainedly broken. Continuing to send traffic at it makes recovery slower and burns your own latency.

**The pattern:**
- Monitor failure rate to a downstream
- When failure rate crosses threshold, **open** the circuit — fast-fail every request without calling downstream
- Periodically allow a probe request through (**half-open**); on success, close the circuit
- Bonus: while open, return a graceful fallback (cached data, default value, queued-for-later)

**AWS specifics:**
- AWS doesn't ship a circuit breaker primitive
- **App Mesh / Service Connect** can do outlier detection at the mesh layer
- For application code: libraries like Polly (.NET), resilience4j (Java), opossum (Node.js)
- **Lambda reserved concurrency** acts as a crude circuit breaker — capping concurrency prevents one downstream's pain from spreading

**When you don't need one:** simple sync request-response with short timeouts and good retry policy often achieves the same effect with less complexity.

## 6. Backpressure

**The problem:** producers send faster than consumers can process. Without backpressure, queues grow unboundedly, memory runs out, latency goes to infinity.

**The pattern:**
- Producers slow down when consumers are saturated
- Manifest as: rejected requests, increased latency, queue length signal

**AWS specifics:**
- **SQS** — queue depth (`ApproximateNumberOfMessagesVisible`) is the backpressure signal; alarm and **slow producers** when it grows
- **Kinesis** — `IncomingRecords` vs shard write throughput; producers handle `ProvisionedThroughputExceeded`
- **Lambda reserved concurrency** — limits concurrent executions; surplus invocations get throttled at the source
- **API Gateway throttling** — per-stage, per-method, or per-API-key (usage plans)
- **DynamoDB on-demand** still has burst limits; provisioned + autoscaling has explicit caps

**The backlog math:** if your queue has N messages and consumer drains at R/sec, recovery takes N/R seconds. Plot it. Set alarms before the queue holds more than you can drain in your error budget window.

**Anti-pattern:** unbounded retries that send retried work back to the same queue, creating a feedback loop. **Reference:** [Avoiding insurmountable queue backlogs (Builders Library)](https://aws.amazon.com/builders-library/avoiding-insurmountable-queue-backlogs/).

## 7. Bulkheads

**The problem:** one component's failure shouldn't take down others. A noisy tenant shouldn't impact other tenants. A busted endpoint shouldn't exhaust shared connection pools.

**The pattern:**
- Partition resources so that exhaustion in one partition doesn't starve others
- Per-tenant Lambda concurrency, per-route DB connection pools, per-customer SQS queues

**AWS specifics:**
- **Lambda reserved concurrency** per function; aliases for further partitioning
- **RDS Proxy** with multiple endpoints to partition connection pools
- **API Gateway usage plans** for per-tenant rate limits
- **Per-tenant queues** in fan-out designs (vs one shared queue) when tenant load varies wildly

## 8. Regional failover

**The problem:** an entire AWS region is degraded or down. Your "five nines" promise is now a "two nines" reality unless you planned for it.

**The pattern:**

| Strategy | RTO | RPO | Cost | When |
|----------|-----|-----|------|------|
| Backup-restore | hours-days | hours | lowest | non-critical, internal tools |
| Pilot light | tens of minutes | minutes | low | important but tolerant of downtime |
| Warm standby | minutes | seconds-minutes | medium | revenue-critical |
| Active-active multi-region | seconds | seconds (sometimes lossy) | highest | mission-critical, contractual |

**AWS specifics:**
- **Route 53 ARC (Application Recovery Controller)** — readiness checks, routing controls, zonal shift
- **Aurora Global Database** — sub-second RPO, ~1min RTO for failover
- **DynamoDB Global Tables** — multi-region active-active; conflict resolution is last-writer-wins
- **S3 Cross-Region Replication** — async; use S3 Multi-Region Access Points for routing
- **Replicate the failover plumbing too** — DKIM keys, secrets, suppression lists, KMS keys

**Reference:** [DR strategies — pilot light / warm standby / multi-site](https://www.factualminds.com/blog/aws-disaster-recovery-strategies-pilot-light-warm-standby-multi-site/) · [Multi-region AWS without doubling costs](https://www.factualminds.com/blog/multi-region-aws-without-doubling-costs/) · [AWS Multi-Region Fundamentals](https://docs.aws.amazon.com/whitepapers/latest/aws-multi-region-fundamentals/aws-multi-region-fundamentals.html).

## 9. Static stability

**The problem:** if your control plane (the part that makes changes) is down, can the data plane (the part serving traffic) keep working?

**The pattern:**
- Workload continues serving traffic with the configuration it has, even if it can't get new configuration
- No dependency on dynamic resolution at request time for things that don't change per request

**AWS specifics:**
- Route 53 health checks fail open (return last-known-good)
- ELB target health is data-plane, not control-plane
- Don't fetch secrets / config on every request — cache and refresh
- IAM evaluates policies in the data plane; the control plane (`UpdatePolicy`) being down doesn't break authorisation

**Reference:** [Static Stability Using Availability Zones (Builders Library)](https://aws.amazon.com/builders-library/static-stability-using-availability-zones/).

## 10. Load shedding

**The problem:** under overload, you get worse service for everyone. Better: get good service for some and explicit failure for the rest.

**The pattern:**
- Detect overload (latency, queue depth, CPU)
- Reject the surplus with a fast 429 / 503 — don't queue it
- Prioritise: shed lowest-priority traffic first
- Pair with a circuit breaker upstream so rejected work doesn't keep coming

**AWS specifics:**
- API Gateway throttling does this at the edge
- Lambda reserved concurrency throttles at the function
- ALB has no native load shedding; do it in your app

**Reference:** [Avoiding overload by putting the smaller service in control (Builders Library)](https://aws.amazon.com/builders-library/avoiding-overload-in-distributed-systems-by-putting-the-smaller-service-in-control/).

## 11. Health checks that mean something

**The problem:** "200 OK on `/health`" doesn't prove the service can serve traffic.

**The pattern:**
- **Shallow health check** — process is alive (used for ELB target health)
- **Deep health check** — service can reach its dependencies (used for self-diagnosis, never for ELB)
- Never make an ELB drop a target because *its* downstream is unhealthy — that escalates a partial outage into a total outage

**Anti-pattern:** `/health` calls the database; database hiccups → all targets marked unhealthy → ELB has no targets → outage.

## 12. Chaos / failure drills

**The problem:** untested DR plans don't work when needed.

**The pattern:**
- Inject failure on a schedule
- Start small (kill a single Lambda invocation), grow into AZ-level and region-level drills
- Track recovery against your stated RTO/RPO

**AWS specifics:**
- **AWS Fault Injection Service (FIS)** — managed chaos experiments
- **Route 53 ARC zonal shift** for AZ-failure drills
- **Aurora failover** — schedulable; do it in non-prod first
